# 台灣/日本鐵道迷網站與App功能調查

調查目的：為台灣鐵道動畫網站（單檔 index.html + Leaflet，無後端，GitHub Pages，時刻表模擬非即時）尋找可借鏡功能。

---

## 1. 鉄道Now（http://www.demap.info/tetsudonow/）

定位：日本全國鐵道「時刻表模擬」運行地圖，Google Maps 疊圖顯示全國列車圖示。

資料方式：**時刻表模擬**（非即時 GPS）——官方明說「鉄道の時刻表情報をもとに現在地を地図上にマッピング」，跟我們的模式完全一致，可視為同類產品的先行者。

亮點功能：
- 列車圖示標示「種別」（普通/快速等）、行進方向、運行狀態（停止/運行中）— 對我們可行性：高，我們已有類似的列車種別/方向呈現，可檢查是否已涵蓋「普通/快速」層級的視覺差異。
- 點擊列車顯示詳細資訊、可指定時間查看該時段運行狀況 — 對我們可行性：中，我們目前無「指定任意時間查看」功能，這點與 Mini Tokyo 3D 的 playback mode 概念重疊（見下）。
- 雙擊列車進入「追蹤模式」，地圖跟隨列車移動 — 對我們可行性：高，我們已有「點車跟隨」，功能對齊。
- 新宿站等轉乘大站會顯示密集列車群聚（被使用者形容為「新宿駅密集しすぎ」而覺得有趣）— 對我們可行性：中，視覺上「車站繁忙感」的呈現可以是加分項，但我們的路網比日本簡單，效果有限。

來源：http://www.demap.info/tetsudonow/ ；https://gigazine.net/gsc_news/en/20121001-tetsudonow/

## 1b. Mini Tokyo 3D（https://minitokyo3d.com ／ GitHub: nagix/mini-tokyo-3d）

定位：東京公共交通即時 3D 地圖（開源專案），結合即時時刻表＋誤點資訊＋3D 地形/建築，鐵道迷與地圖愛好者高度推崇（東京公共交通 Open Data Challenge 得獎作品）。

資料方式：**即時**（時刻表＋官方 Open Data 即時誤點資訊），但同時內建「回放模式」可切換成任意時間點的時刻表模擬——這部分與我們的「無即時資料、純時刻表模擬」高度相關。

亮點功能：
- **回放模式（再生モード）**：可手動選擇任意日期時間，並用滑桿調整播放速度（1x–600x，分段遞增：1-10x 以1為單位、10-100x 以10為單位、100-600x 以100為單位）觀看該時段的列車運行 — 對我們可行性：**高**，這是我們現有「時刻表模擬」架構最自然的延伸功能：使用者可以選一個過去/未來時間點、用倍速快轉看整天路網運作，不需要任何後端或即時資料，純粹是把現有模擬邏輯加上「可調時鐘」介面。
- **列車資訊卡欄位**：路線名、列車種別、行先、列車番號、前站/發車時刻、下一站/到達時刻、延誤時間（有延誤時橙色標示）— 對我們可行性：高，我們已有類似「下一站/ETA」，可以參考其欄位完整度（列車番號、前站發車時刻）補強資訊卡。
- **車站資訊卡含車站照片** — 對我們可行性：中，需要額外圖片素材與版權處理，工作量較大，非零成本。
- **地下模式（地下鉄バージョン）**：地圖整體暗化、地下設施提亮、地上設施半透明，列車進出地下自動切換 — 對我們可行性：低，台灣路網地下段比例與呈現需求不同，且需要精細的地下路線圖資，投入產出比低。
- **日出日落光影變化**：畫面亮度隨真實日出日落時刻變化 — 對我們可行性：中，若用純 CSS/JS 依時刻表對應的當地日出日落時間調整地圖濾鏡亮度，技術上可行且成本低，能增加氛圍感。
- **省電模式（エコモード）**：降低列車與飛機的顯示更新頻率以節省行動裝置電力 — 對我們可行性：中，對行動裝置使用者友善，但目前非急迫需求。
- **飛機顯示、天氣即時疊圖、路線即時搜尋（含誤點後多方案建議）**— 對我們可行性：低，皆需要即時資料源或额外資料集，超出「純前端無後端」的範圍。
- **分享按鈕**（追蹤模式下分享目前列車/航班）— 對我們可行性：高，我們已有深連結，可比對是否涵蓋「追蹤中列車」狀態的分享。
- **多語言介面**（日/英/法/中簡中繁/韓）— 對我們可行性：中，如果目標受眾含國際旅客可考慮，但非核心。

來源：https://minitokyo3d.com/docs/master/ja/user-guide/overview.html ；https://minitokyo3d.com/docs/master/ja/user-guide/display-modes.html ；https://minitokyo3d.com/docs/master/ja/user-guide/screen-and-operations.html ；https://github.com/nagix/mini-tokyo-3d

## 2. 電車走行位置系列（JR西日本 https://www.train-guide.westjr.co.jp/ ／JR東日本「どこトレ」https://doko-train.jp/）

定位：日本各 JR 官方「列車走行位置」頁面，採**線形圖**（路線拉直成一條線，非實際地圖）呈現即時列車位置，是與我們地圖式呈現「互補」的另一種資訊設計典範。

資料方式：即時 GPS/号誌連動資料（官方系統），非模擬。

亮點功能：
- **線形圖呈現**：路線拉直、車站依序排列，列車以箭頭圖示標示方向與位置 — 對我們可行性：中。我們是地圖派，但線形圖適合「單線路線詳情/時刻表視角」的輔助畫面，可考慮做成地圖之外的「路線總覽」次要視圖（例如點選一條路線後跳出該線的線形示意圖，比在地圖上找列車更快掌握全線班距）。
- **列車種別色碼**：普通＝黑、快速＝藍等，依種別上色 — 對我們可行性：高，我們已有復古站名牌配色系統，可比對是否每種列車種別/等級都有一致的色碼區分（尤其台鐵自強/莒光/區間車）。
- **點擊列車資訊卡含車廂級細節**：弱冷房車（弱冷氣車廂）位置、廁所位置、車廂擁擠度 — 對我們可行性：低，這需要車廂配置資料，台灣公開資料難以取得對應細節，且非動畫網站核心價值。
- **誤點分鐘數直接標示在圖示上，15分以上才跳出全域公告**（分級揭露，不干擾正常瀏覽）— 對我們可行性：不適用（我們無誤點資料），但「分級揭露」的介面設計思路可用於別處，例如區分「準點模擬」與「歷史上曾誤點」的路線提示語。
- **どこトレ MYリスト（我的最愛）**：最多收藏5個車站，可加選特定列車，該列車誤點時跳出通知 — 對我們可行性：中，我們已有「最愛收藏」，但「訂閱單一班次的通知」需要背景推播機制，純靜態網站做不到；可行部分是「最愛車站」的收藏數量與呈現方式可以參考其精簡度（5個上限、footer 快速存取）。
- **手機直向優先設計＋底部5鍵固定選單**（MYリスト/駅名檢索/現在地/遲延證明/說明）— 對我們可行性：中，行動版底部固定操作列的資訊架構值得參考，尤其若我們手機版還沒有固定的「常用功能列」。

來源：https://www.train-guide.westjr.co.jp/kyoto.html ；https://www.jr-odekake.net/railroad/service/point/ ；https://trainmark.net/data/column/dokotore-sp.html ；https://doko-train.jp/pc/faq.html

## 3. JR東日本アプリ列車走行位置／Yahoo!乗換案内「トレインキャスト」

定位：官方/大型第三方 App 的列車位置與編成資訊呈現，皆為即時資料。

資料方式：即時（官方運行資料）。

亮點功能：
- **JR東日本アプリ 2023年改版**：列車圖示資訊量增加，一眼可見「種別、行先、混雑状況（車廂擁擠度）」— 對我們可行性：低，擁擠度需要即時載客資料，我們沒有；但「圖示資訊密度」的設計思路（在小圖示上疊加多重資訊而不雜亂）值得參考。
- **改版後被使用者抱怨「資訊太小、老花眼使用者吃力」「視認性反而變差」**— 對我們可行性：這是反面教材，提醒我們加資訊density 時要注意可讀性，行動版尤其要克制。
- **Yahoo!乗換案内「トレインキャスト」(TrainCast，2023年15周年推出)**：全國約700條路線、每日約10萬班列車的動態，依時刻表資料在地圖上即時移動圖示 — 資料規模與呈現方式跟鉄道Now/我們的概念高度相似，差別是它是即時資料，我們是純模擬。對我們可行性：高（其「概念驗證」直接支持我們現有方向），但規模不可比。
- **「動く電車図鑑」設計理念**：新幹線與特急列車圖示採用該列車真實塗裝/車型設計，讓鐵道迷有「看著喜歡的車跑」的收藏感，而非通用箭頭符號 — 對我們可行性：**高**，這點對「鐵道迷取向」網站特別關鍵：如果目前我們的列車圖示是通用符號，改成依車種（自強號 EMU900、普悠瑪 TEMU2000、莒光號等）呈現簡化但可辨識的車型輪廓/配色，會顯著提升鐵道迷的認同感與收藏欲，且純前端 SVG/CSS 就能做，不需要新資料源。
- **點擊列車顯示車次、起訖站、到發時刻、列車番號，並可追蹤特定列車、SNS分享、篩選顯示列車種類**— 對我們可行性：高，我們已有多數對應功能（跟隨、深連結分享），可比對「篩選車種」是否已具備（例如只看自強號、只看區間車）。
- **僅 iOS 版支援 TrainCast，Android 無** — 側面說明這類密集動畫對效能要求高，我們做全平台網頁版需注意行動裝置效能（尤其大量列車同時渲染時的 FPS）。

## 4. 乗り鉄／乗りつぶし記錄文化（レールブック／鉄レコ RailLab／乗りつぶしオンライン）

定位：日本鐵道迷「完乘」文化的記錄工具，核心是把「搭過哪些路線」轉成個人化的地圖/清單/成就系統，社群導向。這類與即時位置無關，是「收藏/紀錄」取向，跟我們現有「最愛收藏」＋「旅程日誌」概念最接近。

資料方式：使用者自行輸入乘車紀錄（非即時資料，純使用者產生內容）。

亮點功能：
- **乗りつぶしオンライン：地圖塗色進度**——日本地圖上未搭乘路段以底色顯示，已搭乘路段「塗上顏色」，隨旅程推進逐步擴大染色範圍；達成率百分比會在完成新路段時「小幅跳動」，使用者形容為「胸のすくような高揚感」（暢快的成就感）— 對我們可行性：**高**，這是無後端也能做的功能：用 localStorage 記錄使用者「看過/搭過」的路線或車站，地圖上對應路段變色，並顯示個人化的「探索進度%」。完全符合我們純前端架構。
- **達成率依系統分類統計**（JR系14%、私鐵系3%等）— 對我們可行性：高，我們有北捷/高捷/中捷/台鐵/高鐵等系統別，可以分系統顯示「已探索車站數/總站數」。
- **排行榜（可選擇公開/不公開參與）**— 對我們可行性：低，需要後端儲存與帳號系統，超出無後端範圍；但可用「分享成果卡片」（產生圖片/連結曬成果）取代排行榜的炫耀動機，純前端可行。
- **鉄レコ（RailLab）雙軌徽章系統**：「完乗バッジ」（完成整條路線/整家公司路網）＋「達成バッジ」（車站數、路線數、車種數、乘車距離的里程碑）— 對我們可行性：高，可設計一套「里程碑徽章」（例如：收藏滿5站、看過3種列車型式、跨兩個系統各看過一班車），用 localStorage 累積成就，不需要伺服器。
- **レールブック**：記錄搭乘區間與日期，並可分享到 Twitter/Facebook — 對我們可行性：中，我們的「旅程日誌」已有停靠/交會/待避事件記錄，若加上「日期戳記＋分享卡片產圖」會更完整，但社群分享圖需要額外的 canvas 產圖邏輯。
- **KR鉄道館「乗りつぶしマップ（JR・全國）」**類個人網站也做類似的塗色地圖，顯示這是鐵道迷普遍偏好的呈現方式，非單一產品孤例 — 對我們可行性：高，多方採用代表這是驗證過的鐵道迷心理機制（收集慾），值得投入。

## 5. 台灣鐵道迷工具總覽

### 5a. Mini Taiwan Project（GitHub: ianlkl11234s/mini-taiwan-learning-project）—— 最直接的「同類競品」

定位：個人 side project，Mapbox GL JS + Three.js 打造的台灣多系統（北捷/高捷/中捷/高鐵/高雄輕軌/台鐵/纜車）即時模擬地圖，跟我們的網站在概念、範圍、技術取徑上幾乎一致，值得當作最主要的對標對象仔細比對差異。

資料方式：時刻表模擬（依真實時刻表模擬列車運行，非即時 GPS）—— 與我們完全同路數。

亮點功能（按重要性排序）：
- **2D/3D 雙模式切換**，3D 模式用 Three.js 把列車渲染成立體方塊、可自由旋轉視角 — 對我們可行性：中，屬於大改動（需要引入 3D 渲染引擎），與我們現有 Leaflet 2D 地圖架構差異大，效益是否值得投入需再評估，非小成本借鏡項目。
- **六段式主題**：Auto／Dawn／Day／Dusk／Night／Dark，可隨真實時間自動切換或手動選 — 對我們可行性：**高**，我們目前只有「明暗主題＋衛星圖」二元切換，比照 Mini Tokyo 3D 的日出日落光影概念，把主題細分成晨昏日夜多階段（純 CSS 濾鏡/色票切換即可），成本低但氛圍感提升明顯。
- **時間控制**：可調整模擬速度 1x–300x、暫停、跳轉到任意時間點 — 對我們可行性：**高**，這與 Mini Tokyo 3D 的「回放模式」概念相同，且對方已用台灣資料實作出可行性證明；我們目前的「放空模式」是自動換車，若加上「手動時間軸＋倍速快轉」會是差異化程度最高、最貼合「時刻表模擬」本質的功能。
- **列車碰撞偵測與視覺標示**：偵測到模擬計算中列車位置重疊/碰撞時，以黃色邊框、警示光暈與位置偏移處理 — 對我們可行性：中，這其實是「模擬演算法的容錯呈現」，如果我們的軌道幾何模擬也可能出現同股列車重疊，這是值得參考的除錯／使用者體驗雙重解法（與我們最近一次 commit「軌道幾何除毛刺」的方向相關）。
- **到站動畫**：列車到站時圓點放大＋脈動效果 — 對我們可行性：高，低成本的微互動，可強化「正在到站」的視覺回饋。
- **依系統分開的統計與趨勢直方圖**（各系統列車數量、時段分布)— 對我們可行性：中，若想呈現「現在路網有多熱鬧」的儀表板小工具，可用現有模擬資料直接算出，不需新資料源。
- **O-D 軌道系統精算站點進度，206 條台鐵軌道、99.8% 覆蓋率** — 對我們可行性：這是資料工程細節而非使用者可見功能，但佐證台鐵路網模擬的軌道覆蓋率是這類專案的重要品質指標，可作為我們自我檢查的參考基準。
- **延伸營運時間軸至隔日 01:30**（跨午夜）— 對我們可行性：已有，我們近期 commit「跨午夜」已涵蓋，功能對齊。
- **五城市（TPE/TXG/KHH/HUN/TTT）快速切換** — 對我們可行性：高，如果我們目前是單一全國地圖畫面，可考慮加「城市/區域快速跳轉」的入口，尤其北中南三大都會捷運系統已支援時更有意義。

### 5b. 其他台灣工具速覽

- **TransTaiwan App「列車即時地圖」**：跟我們同樣用「時刻表＋列車性能＋彎道限速＋誤點狀況」做**估算式**列車位置，且在功能說明中主動揭露「本功能含有相當大的估計成分」與可能失準的原因列表（施工慢行、資料延遲、臨時待避等）— 對我們可行性：**高**，這個「誠實揭露模擬局限」的 UX 寫法直接可用：我們也是模擬而非即時，值得在網站上（例如關於頁或列車卡片角落）加一句類似的誠實聲明，管理使用者期待、建立信任感，屬於零成本的文案改進。來源：https://blog.transtaiwan.com/blog/train-radar-info/
- **鐵道即時通 App**：iPhone 動態島（Dynamic Island）呈現班次狀態/剩餘時間/準點與否，介面比擬航班追蹤 App「Flighty」的專業質感 — 對我們可行性：低（動態島是原生 App 專屬能力，網頁做不到），但「Flighty 式」的極簡資訊卡排版美學可作視覺參考。來源：https://www.koc.com.tw/archives/600688
- **map.qianh.tw／traroutemap（臺灣鐵路路線圖・可達路線網）**：向量設計的全台鐵路路線圖，標示車站等級、整合高鐵/各縣市捷運轉乘資訊，並提供 iOS 捷徑查詢車站代碼、解說列車車次編碼原則 — 對我們可行性：中，「車站等級標示」我們已有（車站分級圖示），但「車次編碼規則科普」是額外的鐵道迷知識型內容，可考慮做成資訊卡片的附加彩蛋（例如自強號車次範圍科普）。來源：https://map.qianh.tw/traroutemap/
- **台灣鐵道通 App**：跨系統（台鐵/高鐵/捷運/輕軌/公車/單車）一頁式行程規劃「我的行程」功能，2022交通部競賽績優獎 — 對我們可行性：低，屬於實用轉乘規劃工具而非鐵道迷動畫欣賞取向，與我們產品定位（觀賞式動畫）不同象限，僅供參考不建議借鏡核心功能。來源：https://play.google.com/store/apps/details?id=com.waccliu.taiwanrail
- **tradiagram.com（台灣鐵路/軌道運行圖）**：從搜尋結果標題判斷是「運行圖/時空圖」風格（時間為X軸、站距為Y軸的斜線圖，類日本「ダイヤグラム」），可依站名/車次查詢 — 對我們可行性：中，頁面本身被 403 擋下無法直接驗證細節（**查無/被擋**，僅能從搜尋摘要判斷其定位），但「運行圖」這個視覺化型態本身是鐵道迷公認的經典呈現方式，可考慮作為地圖之外的「全日班距總覽」次要視圖（呼應 Topic 2 線形圖的类似建議）。來源：https://tradiagram.com/ （403 Forbidden，內容未驗證）
- **一般時刻表 App 共通模式**（火車時刻表／雙鐵時刻表／鐵道時刻）：「我的最愛」收藏常用路線、無廣告簡潔介面被使用者高度重視 — 對我們可行性：高，我們已有最愛收藏功能，可確認是否夠精簡快速（使用者評論特別強調「最少操作迅速查到」）。來源：https://www.dcard.tw/f/railway/p/238777385

## 6. 韓國／中國高鐵動態呈現（餘力調查，非核心）

### 韓國：코레일톡（Korail Talk）「열차 위치 안내」(列車位置服務)

定位：韓國鐵道公社官方 App 內建的即時列車位置功能，2024年韓國鐵道公社「最佳鐵道服務」問卷第一名（22.7%得票、2萬1千人投票），上線5個月累積使用530萬次。

資料方式：即時。

亮點功能：
- 地圖上以「列車種類與番號標示的點」呈現所有運行中列車，提供即時位置＋預估到達時間 — 對我們可行性：不適用（即時資料非我們範疇），但其爆紅的原因值得記錄：**核心賣點只是「不用到車站也能看列車到哪了」的極簡實用性，並非鐵道迷向的美術包裝**，佐證「位置＋ETA」本身就是使用者最在乎的資訊，我們已用模擬方式提供了同款核心體驗，方向正確。
- 使用者對「地圖上不能直接設定起訖站」有抱怨 — 對我們可行性：中，提醒我們檢查目前介面「選定路線/車站」的操作路徑是否夠直覺。

來源：https://www.wikitree.co.kr/articles/1018786 ；https://www.pressian.com/pages/articles/2025011614342377398

### 未進榜但值得記錄的觀察

- **線形圖／運行圖次要視圖**（日本電車走行位置系列的路線拉直呈現、台灣 tradiagram.com 的時空圖概念）：對「單線班距總覽」有價值，但與我們地圖派的核心體驗定位有落差，屬於「若有餘力再做」的次要視圖，非本輪優先。
- **3D 模式（Three.js 立體列車）**：Mini Taiwan Project 已實作，效果吸睛，但技術棧差異大（需引入 3D 引擎），與我們 Leaflet 2D 架構不相容，投入成本遠高於前10項，不建議近期投入。
- **列車碰撞偵測視覺化**：Mini Taiwan Project 的軌道模擬容錯機制，比較像是「除錯輔助工具」而非使用者功能，若我們的軌道幾何模擬（近期才修過「除毛刺」）未來再出現同股重疊問題，可参考其「黃框警示＋位置偏移」做法。
- **韓國 Korail Talk 爆紅案例**：驗證了「位置＋ETA」本身就是使用者最在乎的核心資訊，不需要鐵道迷向的美術包裝也能大受歡迎——間接佐證我們現有的模擬定位方向正確，只是包裝走鐵道迷美學路線。

## 風險與未確定點

- **Mini Taiwan Project 的功能細節**（尤其碰撞偵測、統計直方圖、O-D 覆蓋率數字）僅來自 WebFetch 對 GitHub README 的 AI 摘要整理，未實際開啟專案網站驗證畫面呈現，存在摘要失真風險，若要真的借鏡建議先實際體驗其網站再定案。
- **tradiagram.com、trainvisual.top（中國）、乗りつぶしオンライン 部分頁面** 因 403 Forbidden 被擋，僅能依搜尋摘要判斷定位，未能直接驗證介面細節。
- **koc.com.tw「鐵道即時通」動態島功能**與 Flighty 的類比為單一文章的敘述，未交叉驗證第二來源。
- **韓國/中國段落屬「餘力調查」**，搜尋深度較淺，中國段落實質上未找到鐵道迷取向的獨特視覺化亮點（可能存在但本次搜尋未覆蓋到，例如高德/百度地圖是否有專門的鐵道圖層）。
